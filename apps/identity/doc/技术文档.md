# 技术文档 — Identity 用户中心服务

> **版本**：v1.0 (MVP)
> **最后更新**：2026-04-27
> **技术栈**：NestJS 11 + Prisma 5 + PostgreSQL + Passport-JWT

---

## 1. 服务概述

Identity 服务是 Deer Social 平台的**用户中心**，负责账户生命周期管理、认证、Token 签发与邮箱验证。它是**唯一的 Token 签发方**，Business 服务通过共享 JWT Secret 进行本地校验，无需跨服务调用。

### 核心职责

| 职责 | 说明 |
|------|------|
| 用户注册 | 邮箱注册，bcrypt 哈希密码，注册后立即可用 |
| 用户登录 | 邮箱+密码登录，封禁用户拒绝登录 |
| Token 签发 | 签发 Access Token (2h) + Refresh Token (7d)，通过 HttpOnly Cookie 下发 |
| Token 刷新 | 验证 Refresh Token 后签发新 Token 对 |
| 平台级封禁 | 登录时 + JWT 验证时双重检查 `users.is_banned` |
| 邮箱验证 | 预留验证邮件发送与回调接口（MVP 阶段仅打印日志） |

## 3. 数据库设计

### users 表

Identity 服务**独占写入** `users` 表，其他服务只读。

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | UUID | PK, 默认生成 | 用户唯一标识 |
| `email` | VARCHAR | UNIQUE | 邮箱，用于登录 |
| `password_hash` | VARCHAR | NOT NULL | bcrypt 哈希后的密码（cost=12） |
| `is_verified` | BOOLEAN | 默认 false | 邮箱是否已验证 |
| `is_banned` | BOOLEAN | 默认 false | 平台级封禁标记，封禁后所有业务不可登录 |
| `role` | ENUM | 默认 USER | 用户角色：USER / ADMIN |
| `created_at` | TIMESTAMP | 默认 now() | 创建时间 |
| `deleted_at` | TIMESTAMP | 可空 | 软删除时间，注销后保留 90 天 |

### Prisma Schema

```prisma
model User {
  id            String    @id @default(uuid())
  email         String    @unique
  passwordHash  String    @map("password_hash")
  isVerified    Boolean   @default(false) @map("is_verified")
  isBanned      Boolean   @default(false) @map("is_banned")
  role          UserRole  @default(USER)
  createdAt     DateTime  @default(now()) @map("created_at")
  deletedAt     DateTime? @map("deleted_at")

  @@map("users")
}

enum UserRole {
  USER
  ADMIN
}
```

## 4. API 接口

> 所有接口统一响应格式：
> - 成功：`{ "success": true, "code": 200, "message": "OK", "data": T }`
> - 失败：`{ "success": false, "code": 401, "message": "Unauthorized", "data": null }`

### 4.1 认证接口 (`/auth`)

| 方法 | 路由 | 说明 | 权限 |
|------|------|------|------|
| POST | `/auth/register` | 邮箱注册 | 公开 |
| POST | `/auth/login` | 邮箱登录（Set-Cookie） | 公开 |
| POST | `/auth/refresh` | 刷新 Token 对 | 携带 refresh_token Cookie |
| POST | `/auth/logout` | 登出（清除 Cookie） | 携带 Cookie |
| POST | `/auth/verify-email` | 请求发送验证邮件 | 登录用户 |
| GET | `/auth/verify-callback?token=xxx` | 邮箱验证回调 | 公开 |

### 4.2 用户信息接口 (`/identity`)

| 方法 | 路由 | 说明 | 权限 |
|------|------|------|------|
| GET | `/identity/me` | 获取当前登录用户信息 | 登录用户 |

### 4.3 接口详细说明

#### POST /auth/register

**请求体**：

```json
{
  "email": "user@example.com",
  "password": "123456"
}
```

**响应**：

```json
{
  "success": true,
  "code": 201,
  "message": "OK",
  "data": {
    "id": "632f8576-3a04-4612-8c0f-c13e94dfdc0b",
    "email": "user@example.com",
    "isVerified": false,
    "role": "USER",
    "createdAt": "2026-04-27T01:57:00.000Z"
  }
}
```

**错误**：

| 状态码 | 场景 |
|--------|------|
| 409 | 邮箱已注册 |
| 400 | 参数校验失败 |

#### POST /auth/login

**请求体**：

```json
{
  "email": "user@example.com",
  "password": "123456"
}
```

**响应**：返回用户信息 + 通过 `Set-Cookie` 下发双 Token。

```json
{
  "success": true,
  "code": 201,
  "message": "OK",
  "data": {
    "id": "632f8576-3a04-4612-8c0f-c13e94dfdc0b",
    "email": "user@example.com",
    "isVerified": false,
    "role": "USER"
  }
}
```

**Set-Cookie**：

| Cookie 名 | 属性 | 有效期 |
|------------|------|--------|
| `access_token` | HttpOnly, SameSite=Lax, Secure(生产) | 2 小时 |
| `refresh_token` | HttpOnly, SameSite=Lax, Secure(生产) | 7 天 |

**错误**：

| 状态码 | 场景 |
|--------|------|
| 401 | 邮箱或密码错误 |
| 401 | 账户已被封禁 |

#### POST /auth/refresh

**请求**：携带 Cookie 中的 `refresh_token`。

**响应**：签发新的 Access + Refresh Token，更新 Cookie。

```json
{
  "success": true,
  "code": 201,
  "message": "OK",
  "data": { "message": "Token refreshed" }
}
```

**错误**：

| 状态码 | 场景 |
|--------|------|
| 401 | 缺少 refresh_token Cookie |
| 401 | refresh_token 无效或已过期 |

#### POST /auth/logout

**响应**：清除两个 Cookie。

```json
{
  "success": true,
  "code": 201,
  "message": "OK",
  "data": { "message": "Logged out" }
}
```

#### GET /identity/me

**请求**：携带 Cookie 中的 `access_token`。

**响应**：

```json
{
  "success": true,
  "code": 200,
  "message": "OK",
  "data": {
    "userId": "632f8576-3a04-4612-8c0f-c13e94dfdc0b",
    "email": "user@example.com",
    "isVerified": false,
    "role": "USER"
  }
}
```

**错误**：

| 状态码 | 场景 |
|--------|------|
| 401 | 未登录或 Token 无效 |
| 401 | 账户已被封禁 |

#### POST /auth/verify-email

**请求**：需登录（携带 access_token Cookie）。

**响应**（MVP 阶段仅打印日志，不实际发送邮件）：

```json
{
  "success": true,
  "code": 201,
  "message": "OK",
  "data": { "message": "Verification email sent" }
}
```

#### GET /auth/verify-callback?token=xxx

**响应**：验证成功后更新 `is_verified = true`。

```json
{
  "success": true,
  "code": 201,
  "message": "OK",
  "data": { "message": "Email verified successfully" }
}
```

## 5. 认证与会话

### 5.1 Token 策略

| Token | Payload | 有效期 | 存储 |
|-------|---------|--------|------|
| Access Token | `{ sub, email, isVerified, role }` | 2 小时 | HttpOnly Cookie (`access_token`) |
| Refresh Token | `{ sub }` | 7 天 | HttpOnly Cookie (`refresh_token`) |
| Verify Token | `{ sub, email, purpose: "email-verification" }` | 24 小时 | 不存储，仅用于回调 URL |

### 5.2 登录流程

```
前端 POST /auth/login → Identity
  │
  ▼
验证邮箱 + 密码（bcrypt.compare）
  │
  ├─ 封禁检查（is_banned = true → 401）
  │
  ▼
签发 Access Token + Refresh Token
  │
  ▼
通过 Set-Cookie 下发两个 HttpOnly Cookie
  │
  ▼
返回用户信息 { id, email, isVerified, role }
```

### 5.3 Token 刷新流程

```
前端 POST /auth/refresh → Identity（自动携带 Cookie）
  │
  ▼
从 Cookie 提取 refresh_token
  │
  ├─ 缺失 → 401 Missing refresh token
  │
  ▼
验证 Refresh Token 有效性
  │
  ▼
查询用户是否存在
  │
  ▼
签发新的 Access + Refresh Token，更新 Cookie
```

### 5.4 请求认证流程

```
前端 GET /identity/me → Identity（自动携带 Cookie）
  │
  ▼
JwtAuthGuard → JwtStrategy
  │
  ▼
从 Cookie 提取 access_token
  │
  ▼
验证 Token 签名与有效期
  │
  ▼
查询数据库确认用户存在且未封禁
  │
  ├─ 封禁 → 401 Account has been banned
  │
  ▼
挂载 req.user → 返回用户信息
```

### 5.5 封禁检查机制

封禁在两个环节进行检查：

| 环节 | 检查点 | 行为 |
|------|--------|------|
| 登录时 | `AuthService.login()` | 查询 `users.is_banned`，为 true 则拒绝登录 |
| 每次请求时 | `JwtStrategy.validate()` | 查询数据库 `users.is_banned`，为 true 则拒绝验证 |

## 6. 全局中间件

### 6.1 异常过滤器 (`AllExceptionsFilter`)

捕获所有未处理异常，统一错误响应格式：

| 异常类型 | HTTP 状态码 | 响应消息 |
|----------|------------|----------|
| HttpException | 原始状态码 | 从 `getResponse()` 提取 |
| Prisma P2002 (唯一约束冲突) | 409 | Resource already exists |
| Prisma P2025 (记录不存在) | 404 | Resource not found |
| Prisma 其他错误 | 400 | Database error |
| Prisma 验证错误 | 400 | Validation error |
| 其他 Error | 500 | 原始错误消息 |

### 6.2 响应拦截器 (`ResponseInterceptor`)

统一成功响应格式：`{ success: true, code: number, message: "OK", data: T }`

### 6.3 全局验证管道 (`ValidationPipe`)

- `whitelist: true` — 剥离 DTO 中未声明的属性
- `forbidNonWhitelisted: true` — 存在未声明属性时拒绝请求
- `transform: true` — 自动将请求参数转换为 DTO 指定类型

| 变量名 | 必填 | 说明 | 示例 |

## 9. 与 Business 服务的协作

| 协作项 | 方式 |
|--------|------|
| Token 校验 | Business 服务共享 `JWT_SECRET`，使用本地 `JwtStrategy` 校验，无需调用 Identity |
| 用户信息查询 | Business 服务通过 Prisma Client 只读查询 `users` 表 |
| 平台级封禁 | Business 需要平台级封禁时调用 Identity 内部 API `POST /internal/users/:id/ban` |
| Profile 管理 | 由 Business 服务自行管理 `profiles` 表，Identity 不涉及 |

## 10. 技术要点记录

| 要点 | 说明 |
|------|------|
| NestJS v11 | `APP_FILTER`、`APP_INTERCEPTOR` 等从 `@nestjs/core` 导入（非 `@nestjs/common`） |
| Prisma 版本 | 使用 5.x（7.x 有 breaking changes，schema 不再支持 `url = env()`） |
| Cookie Secure | 开发环境 `secure: false`，生产环境 `secure: true`（通过 `NODE_ENV` 判断） |
| 类型安全 | `jwtService.verify<JwtPayload>()` 泛型避免 `any`；`RequestWithCookies` 接口替代 Express 默认的 `cookies: any` |
| 全局错误 | DTO 校验失败的 `message` 为 `string[]`，过滤器自动 join 为逗号分隔字符串 |
