## 3. 认证与会话

### 3.1 跨服务认证架构

Identity 服务是**唯一的 Token 签发方**，Business 服务**本地校验** Token，避免每次请求都跨服务调用 Identity。

```
┌─────────────┐         ┌─────────────┐         ┌─────────────┐
│    前端      │────────►│  Identity   │         │  Business   │
│             │ Cookie  │  (签发 Token) │         │ (校验 Token) │
└─────────────┘         └─────────────┘         └─────────────┘
                               │                       │
                               │   共享 JWT_SECRET      │
                               │   共享 Prisma Schema   │
                               └───────────────────────┘
```

**关键原则**：
- Identity 与 Business 配置**相同的 `JWT_SECRET`** 环境变量
- Identity 签发 Token 时，Payload 包含 `userId`、`email`、`isVerified`、`role`
- Business 收到请求后，使用本地 `JwtStrategy` 解析并校验 Token，无需网络调用 Identity
- Business 可只读查询 `users` 表获取用户基础信息（代码规范约束不写 users 表）

### 3.2 登录流程（跨服务）

```
前端 POST /auth/login  →  Identity
  │
  ▼
验证邮箱 + 密码
  │
  ▼
生成 Access Token (JWT) + Refresh Token (JWT)
  │
  ▼
通过 Set-Cookie 下发两个 HttpOnly Cookie
  ├─ access_token  (2h Idle Timeout)
  └─ refresh_token (7d 绝对过期)
  │
  ▼
前端 GET /profile/me  →  Business
  │
  ▼
Business 本地校验 Token → 查询 profiles 表
  │
  ▼
返回 { nickname, avatar, mbti, tags, bio }
```

> **要点**：登录成功后前端需要**两次请求**——先 Identity 拿账户凭证，再 Business 拿业务 Profile。Business 的 `/profile/me` 在 Token 有效且 `profiles` 记录不存在时应自动创建默认记录。

### 3.3 Token 刷新

- 前端不感知 Token 过期，每次请求自动携带 Cookie
- Identity 的 `JwtAuthGuard` 校验 `access_token`：
  - 有效 → 继续处理
  - 过期或无效 → 返回 `401 Unauthorized`
- 前端收到 401 后，调用 `POST /auth/refresh`（**请求发往 Identity**）
- Identity 验证 `refresh_token` 后，签发新的 Token 对并通过 `Set-Cookie` 刷新

### 3.4 权限控制

两个服务各自维护 Guard，配置相同但校验逻辑一致：

| Guard | 用途 | 所在服务 |
|-------|------|---------|
| `JwtAuthGuard` | 校验登录状态，提取 `req.user` | Identity + Business |
| `EmailVerifiedGuard` | 校验 `is_verified`，拦截未验证用户访问匹配等敏感功能 | Business |
| `RolesGuard` | 校验 `role` 字段，区分 `USER` / `ADMIN` | Business |
| `BusinessBanGuard` | 校验 `profiles.is_banned`，拦截已被当前业务封禁的用户 | Business |

## 4. 数据库访问层（Prisma）

### 4.1 连接管理

- Neon 为 Serverless PostgreSQL，使用 Prisma 的 `driverAdapters` + `@neondatabase/serverless` 适配器
- 连接池由 Neon 自动管理，后端无需额外配置 PgBouncer
- 事务使用 Prisma `$transaction` API，确保点赞、匹配等复合操作的原子性

### 4.2 关键查询模式

**热度排序（推荐流）**：

```sql
SELECT *,
  (like_count + comment_count * 2) / POWER(EXTRACT(EPOCH FROM (NOW() - created_at)) / 3600 + 2, 1.5) AS score
FROM posts
WHERE status = 'APPROVED'
ORDER BY score DESC
LIMIT 20 OFFSET $1;
```

> MVP 阶段直接在 SQL 中计算，配合 Redis 缓存前 100 条热帖结果（TTL 5 分钟）。

**推荐算法（匹配）**：

```sql
SELECT p.*
FROM profiles p
LEFT JOIN matches m ON (
  (m.user_one_id = $current_user AND m.user_two_id = p.user_id)
  OR (m.user_two_id = $current_user AND m.user_one_id = p.user_id)
)
WHERE p.user_id != $current_user
  AND m.id IS NULL
  AND p.user_id NOT IN (SELECT target_id FROM passes WHERE user_id = $current_user AND created_at > NOW() - INTERVAL '30 days')
ORDER BY (
  array_length(array_intersect(p.tags, $my_tags), 1) * 10
  + COALESCE(p.last_active_score, 0)
) DESC
LIMIT 10;
```

> MVP 阶段使用 SQL + 简单规则引擎，不引入向量数据库。

**操作次数限制逻辑**：

- 每日 00:00 UTC 重置计数
- Redis Key：`daily_actions:{user_id}:{yyyy-mm-dd}`
- 每次 `like` 或 `greet` 时 `INCR`，超过上限（默认 50）则拒绝
- `pass` 不计入上限

**Match 关系建立逻辑**：

```
用户 A like 用户 B
  │
  ▼
查询是否存在 B like A 的记录
  │
  ├─ 存在 → 建立 Match 关系（status: MATCHED），双方开放私聊通道
  └─ 不存在 → 记录 A like B（status: PENDING）
```

**主动发一句话逻辑**：

```
POST /match/greet { targetId, message }
  │
  ▼
检查是否已存在 Conversation
  │
  ├─ 不存在 → 创建 Conversation + 写入第一条 Message
  └─ 存在 → 直接追加 Message
  │
  ▼
返回 conversationId，前端可跳转至聊天页
```

**审计日志自动记录**：

通过 NestJS Interceptor 拦截 Admin Controller 的所有请求，自动写入 `audit_logs` 表：

```typescript
@Injectable()
class AuditInterceptor implements NestInterceptor {
  intercept(context: ExecutionContext, next: CallHandler) {
    const request = context.switchToHttp().getRequest();
    const user = request.user;
    const { method, path, body } = request;
    // 异步写入 audit_logs，不阻塞主响应
    this.auditService.log({ actorId: user.id, action: method + ' ' + path, details: body });
    return next.handle();
  }
}
```

## 6. 实时通信方案

### 6.1 Supabase Realtime 集成

- 不自建 WebSocket 服务，依赖 Supabase 的 Realtime 扩展
- 数据库表 `messages` 启用 Row Level Security + Realtime Broadcast
- 前端订阅：`supabase.channel('messages:conversation_id=eq.xxx').on('INSERT', handler)`
- 后端仅负责写入数据库，推送由 Supabase 自动完成

### 6.2 降级预留

| 场景 | 策略 |
|------|------|
| WebSocket 断连 | 前端自动重连，指数退避（1s → 2s → 4s → 8s，最大 30s） |
| 大规模断连（大陆网络问题） | 纳入技术债务 R-01，二期评估短轮询兜底或迁移至自研 WebSocket |
| 消息丢失 | 用户重新进入聊天室时，前端调用 `GET /chat/conversations/:id/messages?since=lastMessageId` 补发 |

## 7. 关键算法实现

### 7.1 热度排序算法

```typescript
function calculateScore(post: Post): number {
  const hoursSinceCreation = (Date.now() - post.createdAt.getTime()) / (1000 * 60 * 60);
  return (post.likeCount + post.commentCount * 2) / Math.pow(hoursSinceCreation + 2, 1.5);
}
```

- 每小时定时任务或触发器更新热帖缓存
- Redis 缓存 Key：`hot_posts:page:{n}`，TTL 300s

### 7.2 MBTI 维度评分

```typescript
function calculateMBTI(answers: Answer[], level: Level): MBTIResult {
  const dimensions = { E: 0, I: 0, S: 0, N: 0, T: 0, F: 0, J: 0, P: 0 };
  
  for (const ans of answers) {
    const question = questions.find(q => q.id === ans.questionId);
    const weight = question.weights[ans.optionIndex];
    for (const [dim, score] of Object.entries(weight)) {
      dimensions[dim] += score;
    }
  }
  
  const type = [
    dimensions.E >= dimensions.I ? 'E' : 'I',
    dimensions.S >= dimensions.N ? 'S' : 'N',
    dimensions.T >= dimensions.F ? 'T' : 'F',
    dimensions.J >= dimensions.P ? 'J' : 'P',
  ].join('');
  
  return { type, dimensions, level };
}
```

## 8. 安全方案

| 层面 | 措施 |
|------|------|
| 密码存储 | `bcrypt.hash(password, 12)`，禁止明文/MD5 |
| CORS | 限制为前端域名（Vercel 预览环境 + 生产域名） |
| Cookie | `HttpOnly; Secure; SameSite=Lax` |
| 输入校验 | class-validator DTO 校验，防止 SQL 注入与 XSS |
| 敏感词 | 内存 Trie 树或简单 Set 匹配，命中即拒绝 |
| 文件上传 | 限制 MIME 类型（image/jpeg, image/png, image/gif），大小 ≤ 5MB |
| Rate Limiting | MVP 暂不实施，技术债务记录，后续用 `@nestjs/throttler` |
