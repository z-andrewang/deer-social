## 术语表

| 术语 | 定义 |
|------|------|
| **PDB** | Personality Database 的缩写，指以人格类型（如 MBTI）为核心的内容社区模式。 |
| **Profile** | 用户在特定业务模块下的公开身份卡片，包含昵称、头像、标签等信息。 |
| **Match** | 双向喜欢（Mutual Like）后建立的社交关系，双方可进入私聊会话。 |
| **Pass** | 用户在滑动匹配中对某卡片选择左滑，表示不感兴趣。 |
| **金币** | 平台虚拟货币，用于游戏中心内的积分/奖励结算（MVP 阶段仅限游戏场景）。 |
| **自认证身份** | 参考 BlueSky Social 的 AT Protocol 理念，用户关键数据附带密码学签名，确保数据主权与可验证性。 |
| **RTO** | Recovery Time Objective，恢复时间目标。 |
| **RPO** | Recovery Point Objective，恢复点目标。 |

## 文档概述

### 目标

构建一个基于人格标签（PDB 模式）的社区平台，结合滑动匹配社交（Tinder 模式）与 H5 小游戏竞技，实现从"内容讨论"到"关系建立"再到"共同游玩"的用户闭环。

### 范围

- **包含**：Web 端（响应式）用户系统、社区信息流、关注/粉丝关系、滑动匹配、即时私聊、H5 游戏中心、管理后台。
- **不包含**：原生移动 App（iOS/Android）、第三方登录、移动端推送通知、埋点分析系统、支付/提现功能。

### 目标读者

- 前端/后端开发工程师
- 测试工程师
- UI/UX 设计师
- 运维/DevOps 工程师
- 产品经理与项目干系人

## 假设与约束

### 假设

1. 平台初期用户总量不超过 1,000 人，DAU 无明确指标要求。
2. 核心用户群体对 MBTI 及兴趣标签有基本认知。
3. 管理员团队可在工作时间内处理内容审核请求（非 7x24 小时）。

### 技术约束

1. **部署架构**：前后端分离，基于 Monorepo（Turborepo）组织；前端托管于 Vercel，后端部署于 Railway，数据库使用 Neon PostgreSQL，CDN 使用 Cloudflare。
2. **认证方案**：基于 Passport-JWT，Token 存储于 HttpOnly Cookie。
3. **实时通信**：即时消息基于 Supabase Realtime（WebSocket）。
4. **数据存储**：参考 BlueSky Social 的自认证身份理念，用户关键操作数据需具备密码学可验证性；具体签名与验证机制由技术团队在设计阶段细化。

### 业务约束

1. 仅支持邮箱注册登录，不支持第三方 OAuth。
2. 平台初期仅限 Web 端，暂不考虑 App 及推送通知。
3. 游戏中心不实现服务端防作弊，信任客户端上报分数。

## 功能需求

### 5.1 账户中心 (Identity Server)

#### 5.1.1 服务隔离
- **FR-1.1**：用户服务（Identity）与业务服务（社区、匹配、游戏）必须在架构层面隔离，支持独立部署与独立扩展。
- **FR-1.2**：业务服务通过标准 API 向用户服务验证身份与权限，禁止业务服务直接操作用户核心表。

#### 5.1.2 注册与登录
- **FR-1.3**：系统仅支持邮箱注册，不允许第三方登录。
- **FR-1.4**：用户可使用任意邮箱注册，注册后立即可登录并使用基础功能。
- **FR-1.5**：邮箱验证不作为注册强制门槛，但系统需明确区分"已验证"与"未验证"状态。
- **FR-1.6**：用户可在"用户中心"随时发起邮箱验证，系统发送验证邮件，用户点击链接完成验证。
- **FR-1.7**：关键功能（滑动匹配、游戏金币结算、排行榜成绩记录）**仅对已验证邮箱用户开放**；未验证用户尝试访问时，系统需引导其先完成验证。

#### 5.1.3 Profile 管理
- **FR-1.8**：每个用户账户对应且仅对应一个 Profile（取消原"多 Profile"设计）。
- **FR-1.9**：Profile 包含：昵称、头像、MBTI 类型、兴趣标签、个人简介。
- **FR-1.10**：头像与帖子图片上传后存储于 Supabase Storage，需通过 CDN 加速访问。
- **FR-1.11**：用户可修改 Profile 信息，修改后即时生效。

#### 5.1.4 认证与会话
- **FR-1.12**：登录成功后，后端签发 Access Token 与 Refresh Token。
- **FR-1.13**：Access Token 有效期为 48 小时，采用 Idle Timeout 机制（无操作 48 小时后失效）。
- **FR-1.14**：Refresh Token 有效期为 48 小时。
- **FR-1.15**：Token 通过 HttpOnly Cookie 传递，Cookie 需配置 Secure 与标准 SameSite 属性。

### 5.2 社区讨论 (The "PDB" Module)

#### 5.2.1 板块与内容
- **FR-2.1**：社区按人格类型（如 INTP、ENTJ）或兴趣主题划分板块。
- **FR-2.2**：用户可发布图文帖子，支持多图上传。
- **FR-2.3**：帖子发布后即进入"待审核"状态，管理员审核通过后才对外公开显示。
- **FR-2.4**：审核状态包括：`PENDING`（待审）、`APPROVED`（已通过）、`REJECTED`（已拒绝）。被拒绝的帖子仅作者本人可见，并显示拒绝原因。

#### 5.2.2 信息流与排序
- **FR-2.5**：系统提供两条信息流：
  - **推荐流**：基于热度衰减算法排序，公式为 `Score = (点赞数 + 评论数 * 2) / (小时数 + 2)^1.5`。
  - **关注流**：展示用户所关注对象的最新帖子，按时间倒序排列。
- **FR-2.6**：热度排序采用工程简单方案实现，引入 Redis 缓存热帖数据以降低数据库压力。
- **FR-2.7**：MVP 阶段暂不实现置顶帖、精华帖、广告帖等特殊内容类型。

#### 5.2.3 互动
- **FR-2.8**：支持对帖子点赞与取消点赞。
- **FR-2.9**：支持二级评论（即评论下的回复）。
- **FR-2.10**：点赞数与评论数需在帖子详情与列表中实时或准实时反映。

### 5.3 关注与粉丝系统

- **FR-3.1**：用户可关注或取消关注其他用户。
- **FR-3.2**：用户可查看自己的关注列表与粉丝列表。
- **FR-3.3**：关注关系为单向关系，不强制互相关注。
- **FR-3.4**：被关注用户收到通知（应用内通知，非推送）。

### 5.4 匹配社交 (The "Match" Module)

#### 5.4.1 滑动卡片
- **FR-4.1**：系统以卡片形式展示推荐用户，卡片信息包括：昵称、头像、MBTI 类型、兴趣标签、个人简介。
- **FR-4.2**：用户可右滑（Like）或左滑（Pass）。
- **FR-4.3**：每个注册用户每日滑动次数设有上限（具体上限值由运营后台配置，MVP 建议默认 50 次/天），防止恶意刷量。
- **FR-4.4**：被 Pass 的用户在 30 天内不再出现在该用户的推荐池中，30 天后重置。

#### 5.4.2 推荐逻辑
- **FR-4.5**：推荐算法采用简单方案，优先推荐标签重合度高或近期活跃度高的用户。
- **FR-4.6**：技术实现限于后端 SQL 查询或简单规则引擎，MVP 阶段不引入向量数据库或复杂协同过滤。

#### 5.4.3 匹配与主动沟通
- **FR-4.7**：当双方互相 Like 时，系统自动建立 Match 关系，并开放私聊通道。
- **FR-4.8**：用户除被动匹配外，**可直接通过 Profile 页面向任意用户发起私聊**（无需双向 Like）。

### 5.5 即时通讯 (IM)

#### 5.5.1 消息收发
- **FR-5.1**：基于 Supabase Realtime 实现消息实时推送。
- **FR-5.2**：用户进入聊天室后，订阅对应频道，消息发送通过 NestJS API 写入数据库，再由 Realtime 推送给接收方。
- **FR-5.3**：支持文本消息，MVP 阶段暂不支持图片/语音/视频消息。

#### 5.5.2 消息治理
- **FR-5.4**：系统需内置敏感词过滤，命中敏感词的消息禁止发送并提示用户。
- **FR-5.5**：聊天界面需展示防诈骗提示（如首次对话时显示"请注意保护个人隐私与财产安全"）。
- **FR-5.6**：支持消息撤回，撤回后双方均不可见该消息，但后台保留原始记录供审计。

#### 5.5.3 限制与风险
- **FR-5.7**：MVP 阶段不实现离线推送，用户离线时消息仅保存在服务端，上线后通过 Realtime 补发。
- **FR-5.8**：暂不提供轮询兜底方案，依赖 WebSocket 连接稳定性。

### 5.6 游戏中心 (The "Arcade" Module)

#### 5.6.1 游戏接入
- **FR-6.1**：游戏列表通过管理后台动态配置（上架/下架/排序）。
- **FR-6.2**：前端通过 iframe 嵌入 H5 游戏。
- **FR-6.3**：游戏加载流程：
  1. 用户点击游戏卡片，前端调用后端接口获取游戏加载地址；
  2. 后端返回游戏 URL 并附带临时 Token；
  3. 前端通过 iframe 加载该 URL，游戏端凭 Token 向服务端验证并获取当前用户信息（用户ID、昵称、金币余额等）。

#### 5.6.2 游戏结算
- **FR-6.4**：游戏正常结束或异常关闭（如关闭标签页、刷新页面）时，前端需触发结算事件，将本次游戏获得的金币提交至服务端。
- **FR-6.5**：服务端接收结算请求后，校验 Token 有效性，更新用户金币余额。
- **FR-6.6**：MVP 阶段不实现服务端防作弊逻辑，信任客户端上报数据。

#### 5.6.3 排行榜
- **FR-6.7**：每个游戏维护一个全球 Top 100 排行榜。
- **FR-6.8**：排行榜更新允许分钟级延迟，不要求严格实时。
- **FR-6.9**：用户可在排行榜中查看自己的名次与分数。

### 5.7 管理后台 (Admin)

#### 5.7.1 内容审核
- **FR-7.1**：管理员可在后台查看待审核帖子列表，执行通过或拒绝操作。
- **FR-7.2**：管理员可查看用户上传的头像/帖子图片，对违规图片执行下架或标记。

#### 5.7.2 用户管理
- **FR-7.3**：管理员可封禁或解封用户账户，封禁后该用户无法登录。
- **FR-7.4**：管理员可对指定用户清零某款游戏的历史最高分。

#### 5.7.3 游戏配置
- **FR-7.5**：管理员可在后台配置游戏列表，包括游戏名称、加载地址、图标、上架状态。

#### 5.7.4 审计日志
- **FR-7.6**：后台需记录所有管理员的关键操作日志，包括但不限于：内容审核（帖子ID、操作类型、操作人、时间戳）、用户封禁（用户ID、原因、操作人、时间戳）、分数清零（用户ID、游戏ID、操作人、时间戳）。
- **FR-7.7**：审计日志不可被管理员本人修改或删除。

## 非功能需求

### 6.1 性能需求

- **NFR-1**：由于用户规模预期不超过 1,000 人，MVP 阶段不设定严格的 QPS/TPS 指标。
- **NFR-2**：社区推荐流首页加载时间应控制在 2 秒以内（包含网络延迟）。
- **NFR-3**：Redis 用于缓存热帖数据与排行榜数据，减少数据库直接查询压力。

### 6.2 安全需求

- **NFR-4**：用户密码必须经哈希处理后存储（推荐 bcrypt/argon2），禁止明文或简单 MD5 存储。
- **NFR-5**：JWT Token 通过 HttpOnly Cookie 下发，降低 XSS 窃取风险。
- **NFR-6**：所有外部请求需通过 HTTPS 传输。
- **NFR-7**：MVP 阶段暂不实施接口级 Rate Limiting，但需在技术债务清单中记录，待用户增长后优先实施。
- **NFR-8**：用户上传的图片存储于 Supabase Storage，需限制文件类型（仅 JPG/PNG/GIF）与单文件大小（建议不超过 5MB）。
- **NFR-9**：暂不要求上传文件的病毒扫描。

### 6.3 可用性与可靠性

- **NFR-10**：平台面向 Web 端，需支持主流浏览器（Chrome、Firefox、Safari、Edge 最新两个版本）。
- **NFR-11**：响应式布局需覆盖 Mobile（< 768px）、Tablet（768px - 1024px）、PC（> 1024px）三种断点。
- **NFR-12**：Supabase Realtime 的稳定性未在中国大陆网络环境下充分验证，若出现大规模断连，需纳入技术债务并规划后续降级方案（如短轮询）。

### 6.4 数据保护与合规

- **NFR-13**：数据库备份频率为每 4 小时一次全自动备份。
- **NFR-14**：RPO（恢复点目标）为 4 小时，RTO（恢复时间目标）为 2 小时。
- **NFR-15**：用户关键数据（发帖、点赞、匹配记录）参考 BlueSky Social 的自认证身份机制，需支持数据签名与验证，确保平台无法伪造用户操作记录。具体实现方案由技术团队输出设计文档。

## 数据保留与备份策略

| 数据类型 | 保留策略 | 备注 |
|----------|----------|------|
| 用户账户信息 | 永久保留，除非用户主动注销 | 注销后账户标记为软删除，保留 90 天后物理清除 |
| 帖子与评论 | 永久保留，作者删除后进入软删除状态 | 软删除保留 30 天，之后物理清除 |
| 私信记录 | 永久保留 | 用户删除仅影响本地视图，服务端保留 |
| 匹配记录 | 永久保留 | 用于推荐算法优化与审计 |
| 游戏分数 | 永久保留 | 管理员可手动清零 |
| 审计日志 | 永久保留 | 不可删除 |
| 自动备份 | 每 4 小时一次 | 备份保留 7 个周期（约 28 小时滚动） |

## 风险与缓解策略

| 风险ID | 风险描述 | 影响 | 可能性 | 缓解策略 |
|--------|----------|------|--------|----------|
| R-01 | Supabase Realtime 在中国大陆网络环境下出现长连接不稳定或频繁断连 | 高 | 中 | 技术债务记录，若实际发生则评估迁移至自研 WebSocket 或增加短轮询兜底方案 |
| R-02 | 游戏中心无防作弊机制，用户可通过脚本刷金币/分数 | 中 | 高 | MVP 阶段接受该风险，明确在文档中声明"信任客户端"；后续版本引入服务端验算或行为检测 |
| R-03 | 热度排序算法在数据量增长后导致查询性能下降 | 中 | 中 | 已规划 Redis 缓存，后续可引入物化视图或定时任务预计算 |
| R-04 | 用户上传违规图片（色情、暴力）未通过 AI 自动识别，依赖人工审核存在滞后 | 高 | 中 | 管理员加强审核频率；若运营资源允许，后续引入第三方 AI 审核 API |
| R-05 | BlueSky 自认证身份机制与现有 Supabase/PostgreSQL 架构的集成复杂度超预期 | 中 | 中 | 技术负责人需在详细设计阶段输出可行性评估，若成本过高可降级为应用层签名方案 |

## 验收标准

### 账户中心
- [ ] 用户可使用未验证邮箱注册并登录，但进入匹配/游戏结算时被拦截并引导验证。
- [ ] 用户可在个人中心完成邮箱验证，验证后状态实时更新。
