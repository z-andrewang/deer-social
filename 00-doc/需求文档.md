## 术语表

| 术语 | 定义 |
|------|------|
| **PDB** | Personality Database 的缩写，指以人格类型（如 MBTI）为核心的内容社区模式。 |
| **Profile** | 用户在特定业务模块下的公开身份卡片，包含昵称、头像、标签等信息。 |
| **Match** | 双向喜欢（Mutual Like）后建立的社交关系，双方可进入私聊会话。 |
| **Pass** | 用户在匹配中对某卡片选择"不喜欢"，表示不感兴趣，该用户在 30 天内不再出现在推荐池中。 |
| **金币** | 平台虚拟货币，用于游戏中心内的积分/奖励结算（二期功能）。 |
| **MBTI 认证** | 用户通过系统内置测试题答题后获得的 MBTI 结果，附带初级/中级/高级认证标识，区别于用户自行填写的 MBTI 类型。 |
| **RTO** | Recovery Time Objective，恢复时间目标。 |
| **RPO** | Recovery Point Objective，恢复点目标。 |

## 文档概述

### 目标

构建一个基于人格标签（PDB 模式）的社区平台，结合滑动匹配社交（Tinder 模式），实现从"内容讨论"到"关系建立"的用户闭环。H5 小游戏竞技功能规划于二期实现。

### 范围

- **包含（一期 MVP）**：Web 端（响应式）用户系统、社区信息流、关注/粉丝关系、滑动匹配、即时私聊、管理后台。
- **二期规划**：H5 游戏中心（iframe 接入、金币结算、排行榜）。
- **不包含**：原生移动 App（iOS/Android）、第三方登录、移动端推送通知、埋点分析系统、支付/提现功能。

### 目标读者

- 前端/后端开发工程师
- 测试工程师
- UI/UX 设计师
- 运维/DevOps 工程师
- 产品经理与项目干系人

## 假设与约束

### 假设

1. 平台初期用户总量不超过 1,000 人，DAU 无明确指标要求。
2. 核心用户群体对 MBTI 及兴趣标签有基本认知。
3. 管理员团队可在工作时间内处理内容审核请求（非 7x24 小时）。

### 技术约束

1. **部署架构**：前后端分离，基于 Monorepo（Turborepo）组织；前端托管于 Vercel，后端拆分为 **Identity（用户中心）** 与 **Business（业务服务）** 两个独立 NestJS 服务，均部署于 Railway，数据库使用 Neon PostgreSQL，CDN 使用 Cloudflare。
2. **认证方案**：基于 Passport-JWT，Token 由 Identity 服务签发，Business 服务共享 JWT Secret 进行本地校验，Token 存储于 HttpOnly Cookie。
3. **实时通信**：即时消息基于 Supabase Realtime（WebSocket）。
### 业务约束

1. 仅支持邮箱注册登录，不支持第三方 OAuth。
2. 平台初期仅限 Web 端，暂不考虑 App 及推送通知。
3. 游戏中心为二期功能，一期暂不实现。

## 功能需求

### 5.1 账户中心 (Identity Server)

#### 5.1.1 服务隔离
- **FR-1.1**：用户服务（Identity）与业务服务（社区、匹配、游戏）必须在架构层面隔离，支持独立部署与独立扩展。
- **FR-1.2**：业务服务通过标准 API 向用户服务验证身份与权限，禁止业务服务直接操作用户核心表。

#### 5.1.2 注册与登录
- **FR-1.3**：系统仅支持邮箱注册，不允许第三方登录。
- **FR-1.4**：用户可使用任意邮箱注册，注册后立即可登录并使用基础功能。
- **FR-1.5**：邮箱验证不作为注册强制门槛，但系统需明确区分"已验证"与"未验证"状态。
- **FR-1.6**：用户可在"用户中心"随时发起邮箱验证，系统发送验证邮件，用户点击链接完成验证。
- **FR-1.7**：系统保留邮箱验证入口，注册后用户可随时发起验证；MVP 阶段暂不强制拦截未验证用户访问匹配功能（依赖运营阶段根据实际 spam 情况决定是否开启）。

#### 5.1.3 账户核心数据（Identity 独占）
- **FR-1.8**：Identity 服务仅维护账户核心表 `users`，包含：`id`、`email`、`password_hash`、`is_verified`、`is_banned`、`role`、`created_at`。其中 `is_banned` 为**平台级封禁**（如刷号、违法），封禁后该用户在所有业务中均无法登录。
- **FR-1.9**：Identity 服务**不维护 Profile 扩展信息**（昵称、头像、MBTI、标签、简介等），这些信息由业务服务（Business）根据业务场景自行定义和管理。

#### 5.1.4 Profile 管理（Business 业务服务）
- **FR-1.11**：Business 服务维护当前业务的 `profiles` 表，包含：唯一用户名（`username`，用于搜索）、昵称、头像、MBTI 类型（支持自填）、兴趣标签、个人简介、业务级封禁状态（`is_banned`）。`is_banned` 仅限制该用户在当前业务内的操作，不影响其他业务。
- **FR-1.12**：用户可修改 Profile 信息（昵称、头像、用户名、MBTI、标签、简介），修改后即时生效。

#### 5.1.5 认证与会话
- **FR-1.13**：登录成功后，Identity 服务签发 Access Token 与 Refresh Token。
- **FR-1.14**：Access Token 有效期为 2 小时，采用 Idle Timeout 机制（无操作 2 小时后失效）。
- **FR-1.15**：Refresh Token 有效期为 7 天。
- **FR-1.16**：Token 通过 HttpOnly Cookie 传递，Cookie 需配置 Secure 与标准 SameSite 属性。

### 5.2 社区讨论 (The "PDB" Module)

#### 5.2.1 板块与内容
- **FR-2.1**：社区按人格类型（如 INTP、ENTJ）或兴趣主题划分板块。
- **FR-2.2**：用户可发布纯文本帖子。🔜 二期支持图文帖子与多图上传。
- **FR-2.3**：帖子发布后即进入"待审核"状态，管理员审核通过后才对外公开显示。
- **FR-2.4**：审核状态包括：`PENDING`（待审）、`APPROVED`（已通过）、`REJECTED`（已拒绝）。被拒绝的帖子仅作者本人可见，并显示拒绝原因。

#### 5.2.2 信息流与排序
- **FR-2.5**：系统提供两条信息流：
  - **推荐流**：基于热度衰减算法排序，公式为 `Score = (点赞数 + 评论数 * 2) / (小时数 + 2)^1.5`。
  - **关注流**：展示用户所关注对象的最新帖子，按时间倒序排列。
- **FR-2.6**：热度排序采用工程简单方案实现，引入 Redis 缓存热帖数据以降低数据库压力。
- **FR-2.7**：MVP 阶段暂不实现置顶帖、精华帖、广告帖等特殊内容类型。

#### 5.2.3 互动
- **FR-2.8**：支持对帖子点赞与取消点赞。
- **FR-2.9**：支持一级评论。🔜 二期支持二级评论（即评论下的回复）。
- **FR-2.10**：点赞数与评论数需在帖子详情与列表中实时或准实时反映。

### 5.3 匹配社交 (The "Match" Module)

#### 5.3.1 用户卡片与操作
- **FR-4.1**：系统以卡片形式展示推荐用户，卡片信息包括：昵称、头像、MBTI 类型（含自填/认证标识）、兴趣标签、个人简介。
- **FR-4.2**：每张卡片底部提供三个操作按钮：**喜欢**、**不喜欢**、**主动发一句话**。
- **FR-4.2a**：支持通过唯一用户名（`username`）搜索用户并查看其 Profile，搜索到后可直接发起"喜欢"或"主动发一句话"。
- **FR-4.3**：每个注册用户每日有效操作次数设有上限（喜欢 + 主动发一句话合计，MVP 阶段硬编码为 50 次/天），防止恶意刷量。🔜 二期支持运营后台动态配置上限值。
- **FR-4.4**：被点击**不喜欢**的用户在 30 天内不再出现在该用户的推荐池中，30 天后重置。

#### 5.3.2 推荐逻辑
- **FR-4.5**：推荐算法采用简单方案，优先推荐标签重合度高或近期活跃度高的用户。
- **FR-4.6**：技术实现限于后端 SQL 查询或简单规则引擎，MVP 阶段不引入向量数据库或复杂协同过滤。

#### 5.3.3 匹配与主动沟通
- **FR-4.7**：当双方互相点击**喜欢**时，系统自动建立 Match 关系，并开放私聊通道。
- **FR-4.8**：用户点击**主动发一句话**可直接向对方发送一条招呼消息，无需双向 Like；接收方在消息列表中查看后可选择回复或忽略。

### 5.4 即时通讯 (IM)

#### 5.4.1 消息收发
- **FR-5.1**：基于 Supabase Realtime 实现消息实时推送。
- **FR-5.2**：用户进入聊天室后，订阅对应频道，消息发送通过 NestJS API 写入数据库，再由 Realtime 推送给接收方。
- **FR-5.3**：支持文本消息，MVP 阶段暂不支持图片/语音/视频消息。

#### 5.4.2 消息治理
- **FR-5.4**：系统需内置敏感词过滤，命中敏感词的消息禁止发送并提示用户。🔜 MVP 阶段可先依赖用户举报与管理员事后处理，二期实现实时敏感词拦截。
- **FR-5.5**：聊天界面需展示防诈骗提示（如首次对话时显示"请注意保护个人隐私与财产安全"）。
- **FR-5.6**：支持消息撤回，撤回后双方均不可见该消息，但后台保留原始记录供审计。🔜 MVP 阶段暂不支持消息撤回。

#### 5.4.3 限制与风险
- **FR-5.7**：MVP 阶段不实现离线推送，用户离线时消息仅保存在服务端，上线后通过 Realtime 补发。
- **FR-5.8**：暂不提供轮询兜底方案，依赖 WebSocket 连接稳定性。

### 5.5 管理后台 (Admin)

#### 5.5.1 内容审核
- **FR-7.1**：管理员可在后台查看待审核帖子列表，执行通过或拒绝操作。
- **FR-7.2**：管理员可对违规头像执行下架或标记。🔜 二期支持帖子图片审核。

#### 5.5.2 用户管理
- **FR-7.3**：管理员可封禁或解封用户在该业务下的权限（更新 `profiles.is_banned`），封禁后该用户无法在当前业务内操作；平台级封禁（`users.is_banned`）由 Identity 超管操作。

#### 5.5.3 审计日志
- **FR-7.6**：后台需记录所有管理员的关键操作日志，包括但不限于：内容审核（帖子ID、操作类型、操作人、时间戳）、用户封禁（用户ID、原因、操作人、时间戳）。🔜 MVP 阶段可先通过应用日志（stdout）记录，二期实现持久化审计日志表。
- **FR-7.7**：审计日志不可被管理员本人修改或删除。🔜 二期实现。

### 5.6 MBTI 测试与认证

> **MVP 阶段**：仅支持用户在 Profile 中**自填** MBTI 类型，不显示认证标识。测试答题与认证系统 🔜 二期实现。

#### 5.6.1 测试题库
- **FR-8.1**：系统内置 MBTI 测试题库，按难度分为**初级**、**中级**、**高级**三个等级。
- **FR-8.2**：初级测试覆盖基础四维度（E/I、S/N、T/F、J/P），题目数量建议 12~20 题。
- **FR-8.3**：中级测试增加情景题与反向验证题，题目数量建议 30~40 题。
- **FR-8.4**：高级测试引入认知功能栈（Ti、Te、Fi、Fe、Si、Se、Ni、Ne）分析，题目数量建议 50~70 题。

#### 5.6.2 自填与认证双模式
- **FR-8.5**：用户可在 Profile 中**自行选择** MBTI 类型（自填模式），自填结果**不显示任何认证标识**。
- **FR-8.6**：用户通过系统答题完成测试后，系统根据答题结果计算并匹配 MBTI 类型，结果旁显示对应等级认证标识（如 **INTJ · 中级认证**）。
- **FR-8.7**：认证 MBTI 自动覆盖自填 MBTI；用户也可主动清除认证回退到自填模式。

#### 5.6.3 答题与记录
- **FR-8.8**：用户可多次答题，系统保留其获得的**最高等级认证**；若重新答题获得更低等级，不覆盖原高等级认证。
- **FR-8.9**：每次答题生成独立的测试记录，包括答题耗时、各维度得分分布、最终匹配类型。
- **FR-8.10**：用户在答题过程中可中途退出，退出后该次测试不生成结果记录。

## 二期规划

> 以下功能在一期 MVP 中**不实现**，待一期核心功能上线稳定后列入二期迭代计划。

### 6.1 游戏中心 (The "Arcade" Module)

#### 6.1.1 游戏接入
- **FR-6.1**：游戏列表通过管理后台动态配置（上架/下架/排序）。
- **FR-6.2**：前端通过 iframe 嵌入 H5 游戏。
- **FR-6.3**：游戏加载流程：
  1. 用户点击游戏卡片，前端调用后端接口获取游戏加载地址；
  2. 后端返回游戏 URL 并附带临时 Token；
  3. 前端通过 iframe 加载该 URL，游戏端凭 Token 向服务端验证并获取当前用户信息（用户ID、昵称、金币余额等）。

#### 6.1.2 游戏结算
- **FR-6.4**：游戏正常结束或异常关闭（如关闭标签页、刷新页面）时，前端需触发结算事件，将本次游戏获得的金币提交至服务端。
- **FR-6.5**：服务端接收结算请求后，校验 Token 有效性，更新用户金币余额。
- **FR-6.6**：二期初期不实现服务端防作弊逻辑，信任客户端上报数据。

#### 6.1.3 排行榜
- **FR-6.7**：每个游戏维护一个全球 Top 100 排行榜。
- **FR-6.8**：排行榜更新允许分钟级延迟，不要求严格实时。
- **FR-6.9**：用户可在排行榜中查看自己的名次与分数。

#### 6.1.4 管理后台扩展（二期）
- **FR-7.4**：管理员可对指定用户清零某款游戏的历史最高分。
- **FR-7.5**：管理员可在后台配置游戏列表，包括游戏名称、加载地址、图标、上架状态。
- 审计日志扩展：分数清零操作需记录（用户ID、游戏ID、操作人、时间戳）。

### 6.2 关注与粉丝系统

- **FR-3.1**：用户可关注或取消关注其他用户。
- **FR-3.2**：用户可查看自己的关注列表与粉丝列表。
- **FR-3.3**：关注关系为单向关系，不强制互相关注。
- **FR-3.4**：被关注用户收到应用内通知（非推送）。

### 6.3 社区与互动增强

- **FR-2.2**：用户可发布图文帖子，支持多图上传。
- **FR-2.9**：支持二级评论（即评论下的回复）。

### 6.4 即时通讯增强

- **FR-5.4**：系统内置敏感词实时过滤，命中敏感词的消息禁止发送并提示用户。
- **FR-5.6**：支持消息撤回，撤回后双方均不可见该消息，但后台保留原始记录供审计。

### 6.5 管理后台增强

- **FR-4.3**：运营后台可动态配置每日匹配操作次数上限（替代 MVP 硬编码的 50 次/天）。
- **FR-7.2**：管理员可对帖子图片进行审核，对违规图片执行下架或标记。
- **FR-7.6**：后台持久化记录所有管理员的关键操作日志（内容审核、用户封禁、分数清零等）。
- **FR-7.7**：审计日志表不可被管理员修改或删除，支持时间范围筛选与导出。

### 6.6 MBTI 测试与认证系统

- **FR-8.1~FR-8.4**：系统内置初级/中级/高级三套 MBTI 测试题库。
- **FR-8.5~FR-8.7**：自填与认证双模式，认证结果覆盖自填并显示等级徽章。
- **FR-8.8~FR-8.10**：答题记录、最高等级保留、中途退出处理。

### 6.7 账户安全增强

- **FR-1.7**：未验证邮箱用户禁止访问匹配等涉及身份可信度的功能，系统引导其先完成验证。

## 非功能需求

### 7.1 性能需求

- **NFR-1**：由于用户规模预期不超过 1,000 人，MVP 阶段不设定严格的 QPS/TPS 指标。
- **NFR-2**：社区推荐流首页加载时间应控制在 2 秒以内（包含网络延迟）。
- **NFR-3**：Redis 用于缓存热帖数据，减少数据库直接查询压力。

### 7.2 安全需求

- **NFR-4**：用户密码必须经哈希处理后存储（推荐 bcrypt/argon2），禁止明文或简单 MD5 存储。
- **NFR-5**：JWT Token 通过 HttpOnly Cookie 下发，降低 XSS 窃取风险。
- **NFR-6**：所有外部请求需通过 HTTPS 传输。
- **NFR-7**：MVP 阶段暂不实施接口级 Rate Limiting，但需在技术债务清单中记录，待用户增长后优先实施。
- **NFR-8**：用户上传的图片存储于 Supabase Storage，需限制文件类型（仅 JPG/PNG/GIF）与单文件大小（建议不超过 5MB）。
- **NFR-9**：暂不要求上传文件的病毒扫描。

### 7.3 可用性与可靠性

- **NFR-10**：平台面向 Web 端，需支持主流浏览器（Chrome、Firefox、Safari、Edge 最新两个版本）。
- **NFR-11**：响应式布局需覆盖 Mobile（< 768px）、Tablet（768px - 1024px）、PC（> 1024px）三种断点。
- **NFR-12**：Supabase Realtime 的稳定性未在中国大陆网络环境下充分验证，若出现大规模断连，需纳入技术债务并规划后续降级方案（如短轮询）。

### 7.4 数据保护与合规

- **NFR-13**：数据库备份频率为每 4 小时一次全自动备份。
- **NFR-14**：RPO（恢复点目标）为 4 小时，RTO（恢复时间目标）为 2 小时。

## 验收标准

### 账户中心
- [ ] 用户可使用未验证邮箱注册并登录，但进入匹配时被拦截并引导验证。
- [ ] 用户可在个人中心完成邮箱验证，验证后状态实时更新。
- [ ] Access Token 48 小时无操作后自动失效，Refresh Token 可正常续期。

### 社区与互动
- [ ] 帖子发布成功后状态为"待审核"，管理员通过后才出现在推荐流/关注流。
- [ ] 推荐流按热度公式正确排序，关注流按时间倒序展示。
- [ ] 点赞与二级评论功能正常，计数准确。

### 匹配与社交
- [ ] 用户卡片底部展示三个按钮：喜欢、不喜欢、主动发一句话。
- [ ] 用户每日有效操作次数（喜欢 + 主动发一句话）达到上限后，当日无法继续操作。
- [ ] 双向点击**喜欢**后双方自动建立 Match 关系并可私聊。
- [ ] 点击**主动发一句话**可直接向对方发送招呼消息，无需 Match；接收方可回复或忽略。
- [ ] 点击**不喜欢**的用户在 30 天内不再出现在推荐池中，30 天后可能重新出现。

### 即时通讯
- [ ] 消息实时收发延迟不超过 3 秒（正常网络条件下）。
- [ ] 敏感词消息被拦截并提示用户。
- [ ] 消息撤回后双方会话中该消息消失。

### 管理后台
- [ ] 管理员可完成帖子审核、图片审核、用户封禁操作。
- [ ] 所有关键操作均记录审计日志，且不可篡改。

### 安全与可靠性
- [ ] 密码以哈希形式存储，数据库中无明文密码。
- [ ] Cookie 配置 HttpOnly、Secure、SameSite 属性。
- [ ] 数据库备份每 4 小时自动执行，备份文件可成功恢复。
